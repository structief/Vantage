## Context

The secondary sidebar (`SecondarySidebar.tsx`) renders spec names using the raw `slug` value returned by the `/api/repos/.../specs` endpoint. Slugs generated by openspec follow the convention `feature-<name>`, which makes the sidebar list noisy — users see labels like `feature-add-auth` instead of `add-auth`.

Additionally, the sidebar has no awareness of a spec's review status. Status is computed in `SpecDetailView` from `validatedIndices` (local React state) using `deriveStatus(validated, total)` and displayed only within the detail view's title section. No mechanism currently exists to share that value with the sidebar.

The project uses no global state library; shared state is handled via React context (e.g. `SidebarModeProvider`). The same pattern applies here.

## Goals / Non-Goals

**Goals:**
- Strip the `feature-` prefix from spec display labels in the sidebar (routing is unaffected)
- Add a colored status dot next to each spec name in the sidebar
- Make the dot update in real-time as the user validates or unvalidates criteria in the open spec detail view
- Reuse the existing status color mapping (`Draft` → gray, `In review` → amber, `Reviewed` → green) to keep the two surfaces visually consistent

**Non-Goals:**
- Persisting criteria validation state across sessions or page reloads (out of scope; deferred to a future change)
- Showing status dots in collapsed sidebar mode (collapsed mode shows project initials only, not spec lists)
- Fetching or computing status from git/database for specs not open in the current session

## Decisions

### Decision: Shared status via React context
**Choice**: Introduce a `SpecStatusContext` that holds a `Map<slug, SpecStatus>` and exposes an `updateSpecStatus(slug, status)` setter. Wrap the repo layout with a `SpecStatusProvider`. `SpecDetailView` calls `updateSpecStatus` whenever `validatedIndices` changes. `SpecFileList` reads from the context to render each spec's dot.

**Why**: Follows the existing `SidebarModeProvider` pattern — no new dependencies, React-native, synchronous propagation, and zero network overhead. The status is session-only (in-memory), which is acceptable given the "no cross-session persistence" non-goal.

**Alternatives**:
- *Zustand/Jotai*: Unnecessary weight for a single shared value; the project's constitution prefers a slim runtime.
- *SQLite write-on-toggle + sidebar fetch*: Too heavy; introduces latency and requires a new API endpoint for status reads.
- *URL search param or localStorage*: Awkward for a value that changes on every checkbox toggle; not appropriate for transient UI state.

### Decision: Prefix stripping in the display layer only
**Choice**: Strip the `feature-` prefix inside `SpecFileList` at render time using a simple string replacement (`slug.replace(/^feature-/, "")`). The raw slug is still used for `href` construction and as the React `key`.

**Why**: The prefix is a file-naming convention, not a user-facing identity. Stripping it only in the display label keeps the change minimal and non-breaking — no API changes, no data model changes.

**Alternatives**:
- *Strip in the API response*: Would require the UI to separately track the full slug for routing, complicating the data shape for no benefit.

## Data model changes

None. Status is held in-memory in React context for the current session only.

## API changes

None.

## Risks / Trade-offs

- [Status resets on navigation] If the user navigates away from the repo and back, `validatedIndices` and the context map are reset, so all dots revert to Draft. → Acceptable given the no-persistence non-goal; a future persistence change will address this.
- [Context provider scope] The provider must be placed at the repo layout level (wrapping both the sidebar and the spec page) to allow both consumers to share the same instance. If placed too high (app root), it wastes memory; too low, the sidebar and detail view get separate instances. → Mount at `app/repo/[owner]/[name]/layout.tsx`.

## Open Questions

- Should the status dot be shown for specs that are currently active (highlighted) in the sidebar? Yes — it should always be visible regardless of active state, consistent with how it appears in the spec title section.
